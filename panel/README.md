# Aras-GP Panel

پنل مدیریت تحت وب برای موتور رله‌ی این پروژه (عبور از فیلترینگ با روش
Domain Fronting از طریق Google Apps Script + Cloudflare Workers).

پنل **موتور رله را بازنویسی نمی‌کند**. یک لایه‌ی Flask روی همان کد موجود در
`src/` است که آن را پیکربندی، اجرا، پایش و دیپلوی می‌کند.

---

## اجرا

```bash
python3 -m venv .venv                # ویندوز: py -m venv .venv
source .venv/bin/activate            # ویندوز: .venv\Scripts\activate
pip install -r requirements.txt -r panel/requirements.txt

python -m panel
```

روی macOS دستور `python` وجود ندارد؛ برای ساخت venv حتماً `python3` بزنید.
پس از فعال‌شدن venv، `python` داخل آن در دسترس است.

سپس <http://127.0.0.1:8600> را باز کنید. در اولین اجرا یک رمز عبور برای پنل
تعیین می‌کنید (با PBKDF2 هش می‌شود).

متغیرهای محیطی:

| متغیر | پیش‌فرض | توضیح |
|---|---|---|
| `ARAS_PANEL_HOST` | `127.0.0.1` | آدرس شنود پنل |
| `ARAS_PANEL_PORT` | `8600` | پورت پنل |
| `ARAS_DATA_DIR` | `panel/data` | محل نگهداری وضعیت پنل |
| `DFT_CONFIG` | `config.json` | مسیر کانفیگ رله (همان متغیر `main.py`) |
| `ARAS_LICENSE_PUBKEY` | — | فعال‌سازی قفل لایسنس (پایین را ببینید) |

### پیش‌نیازها

* Python 3.10+
* یک حساب Cloudflare با API Token دارای دسترسی **Account → Workers Scripts → Edit**
  (فقط برای دیپلوی خودکار Worker)
* یک حساب Google برای استقرار Apps Script

## معماری: پنل چطور به رله وصل می‌شود؟

**روش انتخاب‌شده: ترد داخلی با event loop اختصاصی (نه subprocess).**

رله یک برنامه‌ی asyncio است و Flask یک برنامه‌ی همگام WSGI؛ این دو نمی‌توانند
یک event loop مشترک داشته باشند. `panel/relay_manager.py` یک ترد جداگانه
می‌سازد، داخل آن یک event loop تازه راه می‌اندازد و شیء `ProxyServer` را
همان‌جا می‌سازد. پنل یک ارجاع به همان شیء نگه می‌دارد.

چرا این روش و نه subprocess کردن `main.py`؟ چون داشبورد باید **زنده** باشد:
`fronter.stats_snapshot()` و جدول حساب کاربران مستقیماً از شیء در حال اجرا
خوانده می‌شوند، نه از روی pars کردن خروجی لاگ.

هر فراخوانی که به وضعیت رله دست می‌زند با `run_coroutine_threadsafe` به همان
event loop رله برگردانده می‌شود؛ بنابراین دیکشنری‌های داخلی رله فقط از تردی
خوانده می‌شوند که آن‌ها را تغییر می‌دهد و نیازی به قفل نیست.

**هزینه‌ی این انتخاب (صادقانه):** پنل و رله در یک پروسه‌اند، پس یک کرش
مدیریت‌نشده هر دو را می‌برد و توقف رله حافظه را مثل kill کردن یک subprocess
آزاد نمی‌کند. در عوض داشبورد واقعاً زنده است.

## گزینه‌های انتخاب‌شده

پرامپت اصلی برای دو بخش، دو مسیر ممکن تعریف کرده بود. این‌ها انتخاب شد:

### Google Apps Script → **گزینه‌ی ساده**

پنل `Code.gs` را با `AUTH_KEY` و `WORKER_URL` شما تولید می‌کند، دکمه‌ی کپی و
راهنمای هفت‌مرحله‌ای می‌دهد، و شما Deployment ID را برمی‌گردانید.

دیپلوی کامل از طریق API پیاده **نشده**، و دلیلش را پنهان نکرده‌ایم: به یک
OAuth2 Client ثبت‌شده در Google Cloud، فعال‌سازی Apps Script API روی حساب
کاربر نهایی، و انتشار صفحه‌ی رضایت نیاز دارد. آن مسیر برای کاربر عادی
سنگین‌تر از انجام دستی همان شش مرحله است. محل اضافه‌کردنش در آینده،
`panel/gasgen.py` است (یک تابع `deploy_via_api()` کنار `render()`).

### کاربران → **گزینه‌ی کامل**

فقط UI ظاهری نیست؛ یک لایه‌ی احراز هویت واقعی به موتور اضافه شده است:

* `src/account_manager.py` (فایل جدید) — جدول حساب‌ها، هش PBKDF2، سهمیه،
  تاریخ انقضا، شمارنده‌ی بایت، و wrapperهای شفاف روی
  StreamReader/StreamWriter برای شمارش ترافیک.
* `src/proxy_server.py` (تغییر یافته) — `Proxy-Authorization: Basic` روی
  پروکسی HTTP (RFC 7235) و نام‌کاربری/رمز روی SOCKS5 (RFC 1929)، به‌علاوه
  یک sweeper که هر ۵ ثانیه حساب‌های ازسهمیه‌گذشته یا منقضی را قطع می‌کند.

نکات مهم:

* **کاملاً اختیاری است.** بدون `proxy_auth.enabled` در `config.json`،
  `AccountManager.required` برابر `False` است و پروکسی دقیقاً مثل قبل، بدون
  رمز کار می‌کند. رفتار تک‌کاربره‌ی اصلی پروژه دست‌نخورده باقی می‌ماند.
* رمزها هرگز به‌صورت متن ساده ذخیره نمی‌شوند (PBKDF2-HMAC-SHA256، ۱۲۰٬۰۰۰ دور).
  چون کلاینت‌های پروکسی در هر اتصال دوباره احراز هویت می‌کنند، یک رمز
  تأییدشده به‌صورت HMAC در حافظه cache می‌شود تا KDF به‌ازای هر اتصال اجرا نشود.
* کاربران در `config.json` زیر کلید `proxy_auth` ذخیره می‌شوند تا رله
  بتواند با همان جدول از `main.py` هم بالا بیاید — یک منبع حقیقت، نه دو تا.
* ویرایش کاربران بلافاصله روی رله‌ی در حال اجرا اعمال می‌شود
  (`RelayManager.sync_accounts`)؛ نیازی به ری‌استارت نیست.

کلید جدید در `config.json` (بقیه‌ی کلیدها دست‌نخورده‌اند):

```jsonc
"proxy_auth": {
  "enabled": false,
  "realm": "Aras-GP",
  "users": [
    { "username": "ali", "salt": "…", "hash": "…", "iterations": 120000,
      "quota_bytes": 2147483648, "expires_at": "2026-12-31",
      "enabled": true, "up_bytes": 0, "down_bytes": 0 }
  ]
}
```

## بخش‌های پنل

| صفحه | کار |
|---|---|
| **داشبورد** | آمار زنده از `stats_snapshot()`، نمودار سرعت (پولینگ ۵ ثانیه‌ای از `/api/stats`)، جدول پرترافیک‌ترین میزبان‌ها |
| **وضعیت** | اجرای رله، آخرین اتصال موفق، اعتماد سیستم به CA (با `is_ca_trusted`)، لاگ زنده |
| **دیپلوی** | آپلود واقعی Worker به Cloudflare + تولید `Code.gs` و ثبت Deployment ID |
| **ساخت کانفیگ** | فرم کامل تمام کلیدهای `config.example.json` + تولید `auth_key` تصادفی + پروفایل‌ها |
| **کاربران** | مدیریت حساب‌های پروکسی، سهمیه، انقضا، قطع اتصال |
| **تنظیمات** | رفتار پنل، رمز پنل، ویرایش `.env` فورواردر، وضعیت لایسنس، شفافیت شبکه |

### دیپلوی Cloudflare — دقیقاً چه کاری می‌کند

1. `GET /user/tokens/verify` — اعتبارسنجی توکن
2. `GET /accounts/{id}/workers/subdomain` — یافتن زیردامنه‌ی workers.dev
3. جایگزینی `WORKER_URL` داخل `worker.js` با نام میزبان واقعی (محافظ حلقه)
4. `PUT /accounts/{id}/workers/scripts/{name}` — آپلود multipart به‌صورت ES module،
   به‌همراه بایند `UPSTREAM_FORWARDER_URL` اگر پر شده باشد
5. `POST /accounts/{id}/workers/scripts/{name}/subdomain` — فعال‌سازی workers.dev

## امنیت

* پنل پشت لاگین است (PBKDF2). کوکی نشست `HttpOnly` + `SameSite=Lax`، انقضای ۱۲ ساعت.
* توکن CSRF روی همه‌ی فرم‌ها و درخواست‌های غیر-GET.
* Rate limit روی لاگین (۸ تلاش / ۵ دقیقه)، دیپلوی (۶ / ۵ دقیقه)، تست اتصال و نصب CA.
  فقط تلاش‌ها شمرده می‌شوند، نه بازدید صفحه — وگرنه کاربر از صفحه‌ی لاگین خودش قفل می‌شد.
* CSP سخت‌گیرانه: `default-src 'self'` و `script-src 'self' 'nonce-…'`.
  هیچ `unsafe-inline` برای اسکریپت وجود ندارد؛ هر بلاک inline یک nonce یکبارمصرف دارد.
* `config.json`، `panel/data/*` و `.env` فورواردر با دسترسی `0600` و به‌صورت
  اتمیک نوشته می‌شوند. `panel/data/` در `.gitignore` است.
* توکن Cloudflare **به‌صورت پیش‌فرض ذخیره نمی‌شود**؛ فقط با تیک صریح کاربر.
* هیچ مقدار محرمانه‌ای لاگ نمی‌شود (`security.scrub` / `security.redact`).
* بخش «سلب مسئولیت» README در فوتر همه‌ی صفحات باقی مانده است.

### هشدار درباره‌ی باز کردن پنل روی شبکه

پنل `auth_key` رله و (در صورت ذخیره) توکن Cloudflare را در اختیار دارد.
پیش‌فرض آن `127.0.0.1` است. اگر `ARAS_PANEL_HOST` را تغییر دادید، حتماً پشت
VPN یا یک reverse proxy با TLS و احراز هویت قرارش دهید. پنل هنگام bind شدن
روی آدرس غیر-loopback این هشدار را در لاگ چاپ می‌کند.

## شفافیت شبکه (مهم)

این پروژه ابزار عبور از سانسور است. بنابراین پنل عمداً فاقد موارد زیر است:

* **بدون تله‌متری / phone-home.** هیچ IP، آمار ترافیک یا هویتی به هیچ سرور
  مرکزی فرستاده نمی‌شود.
* **بدون کیل‌سوییچ از راه دور.** هیچ کانالی برای غیرفعال‌کردن رله‌ی کاربر وجود ندارد.
* **بدون کد مبهم‌سازی‌شده.** همه‌چیز قابل حسابرسی است.
* **بدون CDN خارجی.** فونت، آیکون، CSS و JS همگی محلی‌اند. نه Tailwind CDN،
  نه Font Awesome، نه Google Fonts، نه کتابخانه‌ی نمودار. دلیل: یک درخواست به
  CDN، خودِ اجرای این ابزار را لو می‌دهد و پنل روی شبکه‌ی فیلترشده هم باید
  کامل رندر شود.

فهرست کامل مقصدهای خروجی این پروسه:

1. `api.cloudflare.com` — فقط هنگام بررسی توکن یا دیپلوی Worker
2. هرچه خود موتور رله با آن حرف می‌زند (Apps Script و Worker **خود شما**)

همین. این فهرست در صفحه‌ی «تنظیمات» هم به کاربر نشان داده می‌شود.

> **دلیل طراحی:** هر سازوکار پنهانِ ارتباط با یک سرور مرکزی، همان ریسک
> نظارتی و «نقطه‌ی شکست متمرکز» را برای کاربران نهایی می‌سازد که این پروژه
> قرار است از آن محافظت کند. اگر چنین سروری روزی هک یا توقیف شود، هم کاربران
> و هم اعتبار پروژه در خطر است.

## لایسنس

* پوشه‌ی `panel/` تحت پروانه‌ی اختصاصی `panel/LICENSE` است: حذف/تغییر
  برندینگ، ری‌برندسازی و فروش مجدد بدون اجازه‌ی کتبی ممنوع است — ولی
  حسابرسی امنیتی، انتشار نتایج بررسی، و اجرای خصوصی نامحدود آزاد است.
* موتور رله (`src/`, `deploy/`, `main.py`, `setup.py`) — از جمله
  `src/account_manager.py` و تغییرات `src/proxy_server.py` — همچنان تحت
  **MIT** فایل `LICENSE` ریشه است، چون از کد MIT مشتق شده‌اند. پروانه‌ی پنل
  آن اجازه را پس نمی‌گیرد.

### قفل لایسنس آفلاین (اختیاری، پیش‌فرض خاموش)

بررسی کاملاً محلی است: یک امضای Ed25519. **هیچ تماس شبکه‌ای، هیچ فعال‌سازی
از راه دور.** بدون کلید عمومی، قفل غیرفعال است و پنل عادی بالا می‌آید.

```bash
# ۱) ساخت کلید امضا (فقط نزد فروشنده می‌ماند)
python -m panel.licensing keygen --out vendor-key.pem

# ۲) کلید عمومی چاپ‌شده را در panel/licensing.py قرار دهید
#    (یا متغیر محیطی ARAS_LICENSE_PUBKEY)

# ۳) صدور لایسنس برای یک مشتری
python -m panel.licensing sign --key vendor-key.pem \
    --licensee "شرکت نمونه" --days 365

# ۴) بررسی
python -m panel.licensing verify --file panel/data/license.key
```

## سلب مسئولیت

این نرم‌افزار فقط برای اهداف آموزشی، تحقیقاتی و تست ارائه شده است.

- نرم‌افزار «همان‌طور که هست» (AS IS) ارائه می‌شود، بدون هیچ ضمانتی.
- توسعه‌دهندگان مسئولیتی در قبال خسارات احتمالی ندارند.
- رعایت قوانین محلی، ملی و بین‌المللی بر عهده‌ی کاربر است.
- رعایت شرایط استفاده از سرویس‌های Google و Cloudflare بر عهده‌ی کاربر است.
